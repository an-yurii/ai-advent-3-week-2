**Название:** AI‑агент с веб‑интерфейсом на Go + GigaChat

**Цель:** разработать клиент‑серверное приложение на Go с AI‑агентом для взаимодействия с GigaChat API. Приложение должно предоставлять веб‑интерфейс для общения с моделью, хранить историю диалога в памяти и поддерживать параллельную обработку запросов.

#### Требования

1. **Архитектура:**
    * клиент‑серверное приложение на Go (актуальная стабильная версия);
    * AI‑агент — отдельная сущность с инкапсулированной логикой;
    * веб‑интерфейс для взаимодействия с пользователем;
    * развёртывание через Docker (актуальная версия) с `docker-compose.yml`.

2. **Функционал:**
    * приём пользовательского запроса через веб‑интерфейс;
    * отправка запроса в GigaChat через актуальный эндпоинт;
    * получение ответа от модели;
    * возврат ответа пользователю в веб‑интерфейсе;
    * хранение истории диалога в памяти (для каждого пользователя отдельно);
    * параллельная обработка запросов от разных пользователей.

3. **Конфигурация:**
    * API‑ключ GigaChat передаётся через переменную окружения `GIGACHAT_API_KEY`;
    * без аутентификации;
    * без HTTPS.

4. **Логирование:**
    * логирование в консоль;
    * формат: `[дата-время с точностью до миллисекунд] [тип] [сообщение]`;
    * логировать:
        * входящие запросы (заголовки и тело);
        * исходящие запросы к GigaChat (заголовки и тело);
        * ответы от GigaChat (заголовки и тело);
        * ошибки.

5. **Обработка ошибок:**
    * при недоступности GigaChat API — возвращать ошибку с кодом $503$;
    * при некорректных запросах от клиента — возвращать ошибку с кодом $400$;
    * все ошибки логировать.

6. **Тестирование:**
    * юнит‑тесты для ключевых компонентов (агент, обработчики HTTP);
    * интеграционные тесты для сквозного сценария (запрос → GigaChat → ответ).

7. **Документация:**
    * описание API сервера (эндпоинты, методы, форматы запросов/ответов);
    * инструкция по развёртыванию и настройке Docker‑контейнеров.

#### Техническое описание

**Компоненты системы:**

1. **HTTP‑сервер** (Go):
    * обрабатывает входящие запросы от веб‑интерфейса;
    * управляет сессиями пользователей (идентификация по ID сессии);
    * передаёт запросы AI‑агенту.

2. **AI‑агент** (отдельная сущность в Go):
    * инкапсулирует логику взаимодействия с GigaChat;
    * хранит историю диалога для каждого пользователя в памяти (структура `map[sessionID][]Message`);
    * формирует запрос к GigaChat с учётом истории;
    * отправляет запрос через актуальный эндпоинт GigaChat API;
    * получает и парсит ответ;
    * возвращает результат серверу.

3. **Веб‑интерфейс** (статические файлы, обслуживаемые сервером):
    * простая HTML‑страница с полем ввода и областью вывода;
    * JavaScript для отправки запросов и отображения ответов.

4. **Docker‑окружение:**
    * контейнер для Go‑приложения;
    * `docker-compose.yml` для запуска.

**Эндпоинты сервера:**

* `GET /` — отдаёт веб‑интерфейс (HTML);
* `POST /api/chat` — обрабатывает запросы пользователя:
    * вход: JSON `{ "message": "текст запроса", "session_id": "ID сессии" }`;
    * выход: JSON `{ "response": "ответ модели", "session_id": "ID сессии" }`.

**Структура данных для истории диалога:**
```go
type Message struct {
    Role    string `json:"role"` // "user" или "assistant"
    Content string `json:"content"`
}

type Session struct {
    ID      string
    History []Message
}
```

#### План реализации

**Шаг 1. Настройка проекта и Docker**
* создать структуру проекта;
* настроить `Dockerfile` для Go‑приложения;
* написать `docker-compose.yml` с сервисом приложения.

**Шаг 2. Реализация HTTP‑сервера**
* настроить маршрутизацию (`/` и `/api/chat`);
* реализовать обработчик для `/api/chat`:
    * парсинг JSON‑запроса;
    * валидация полей (`message`, `session_id`);
    * вызов AI‑агента;
    * формирование JSON‑ответа;
    * обработка ошибок (400, 503).

**Шаг 3. Реализация AI‑агента**
* определить структуру агента с хранилищем истории (`map[string][]Message`);
* реализовать метод `SendMessage(sessionID string, userMessage string) (string, error)`:
    * получить историю для `sessionID`;
    * сформировать payload для GigaChat (включая историю);
    * отправить POST‑запрос к актуальному эндпоинту GigaChat;
    * обработать ответ (извлечь `content` из `choices[0].message`);
    * добавить запрос и ответ в историю;
    * вернуть текст ответа.

**Шаг 4. Веб‑интерфейс**
* создать `index.html` с формой ввода и областью вывода;
* добавить JavaScript для:
    * генерации `session_id` (UUID);
    * отправки POST‑запроса на `/api/chat`;
    * отображения ответа.

**Шаг 5. Логирование**
* использовать пакет `log/slog` с форматом времени `2006-01-02 15:04:05.000`;
* логировать:
    * старт сервера;
    * входящие HTTP‑запросы (метод, путь, заголовки, тело);
    * запросы к GigaChat (URL, заголовки, тело);
    * ответы от GigaChat (статус, заголовки, тело);
    * ошибки (с трассировкой).

**Шаг 6. Тестирование**
* юнит‑тесты:
    * проверка валидации входных данных;
    * проверка формирования payload для GigaChat;
    * проверка парсинга ответа от GigaChat.
* интеграционные тесты:
    * запуск сервера в тестовом режиме;
    * имитация HTTP‑запроса от клиента;
    * проверка ответа сервера и корректности истории.

**Шаг 7. Документация**
* **API.md**:
    * эндпоинты (`GET /`, `POST /api/chat`);
    * примеры запросов/ответов;
    * коды ошибок.
* **README.md**:
    * инструкция по запуску:
        * установка Docker;
        * создание `.env` с `GIGACHAT_API_KEY`;
        * запуск через `docker-compose up`.

#### Пример конфигурации

**`docker-compose.yml`**
```yaml
version: '3.8'
services:
  app:
    build: .
    ports:
      - "8080:8080"
    environment:
      - GIGACHAT_API_KEY=${GIGACHAT_API_KEY}
    volumes:
      - ./static:/app/static
```

**`Dockerfile`**
```dockerfile
FROM golang:1.21 AS builder
WORKDIR /app
COPY . .
RUN go mod download
RUN CGO_ENABLED=0 GOOS=linux go build -o app .

FROM alpine:latest
RUN apk --no-cache add ca-certificates
WORKDIR /root/
COPY --from=builder /app/app .
EXPOSE 8080
CMD ["./app"]
```
